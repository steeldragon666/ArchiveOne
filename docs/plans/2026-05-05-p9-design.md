## Architecture

P9 delivers commercial-readiness + partnership-extensibility across **four sequenced phases / four PRs**, with Phase 1 as the revenue gate. Each phase ships independently; later phases are deferrable without breaking earlier ones.

**Phase shape:**

```
Week:    1    2    3    4    5    6    7    8    9
         │    │    │    │    │    │    │    │    │
P9.0: ████ Phase 0 — production deployment (GCP)
       Cloud Run + Cloud SQL + Secret Manager + DNS + monitoring wired
                       ▼
                       ✅ Production environment lives
                       │
P9.1:        ████████████ Phase 1 — billing LIVE
              Stripe + GST + plan model + Checkout-in-onboarding +
              webhook handlers + tenant activation gating
                                       ▼
                                       ✅ FIRST CUSTOMER CAN PAY (revenue gate)
                                       │
P9.2:                              ████████ Phase 2 — billing operations
                                       Plan tiers + Customer Portal +
                                       dunning + invoice history +
                                       per-claim metering + mobile bulk discount
                                                          ▼
                                                          ✅ SaaS-grade billing
                                                          │
P9.3:                                                ███████ Phase 3 — federation primitives
                                                          delegation_token API +
                                                          brand JSON + view-as-financier
                                                          mode + token redemption flow
                                                                              ▼
                                                                              ✅ Financier-demo capability
```

**Why this shape:**

1. **Phase 0 first**: hosting choice (GCP) needs to land before P9.1's Stripe webhook handler can register a stable production URL. Migrating hosts mid-billing would mean re-registering webhooks and risking split-brain (Stripe sends to old URL, new deployment misses events).
2. **Phase 1 = revenue critical path**: at end of Phase 1, the customer-with-MSA can pay you and use the product. If anything else slips, this gate is preserved.
3. **Phase 2 = operational depth**: polish that turns billing from "live" into "SaaS-grade." Gets focused attention without revenue pressure.
4. **Phase 3 = speculative-investing scoped tight**: builds generic federation primitives that get re-used when a real financier appears. The minimal generic UI doubles as a sales tool for ongoing financier conversations.

**Estimate:** ~9 weeks total. Phase 0 ~1 week, Phase 1 ~3-4 weeks, Phase 2 ~2-3 weeks, Phase 3 ~2-3 weeks.

## Pricing model — LOCKED

Critical context for the data model and Stripe integration.

| Component | Stripe model | Detail |
|-----------|--------------|--------|
| **Trial period** | One-time, free | 30 days from signup; 2 seats max, 5 claimants max; mobile included for trial claimants; basic branding (firm name + logo only); no federation; read-only access for 30 days post-expiry then archive |
| **Onboarding fee** | One-time invoice | AUD **$5,000** at trial → paid conversion |
| **Per-claim platform fee** | Metered usage record | **$1,500 per delivery** — a delivery is either a quarterly assurance update OR an annual claim. Firms doing quarterly mode pay 4× $1,500 = $6,000/claimant/year on per-claim fees alone. |
| **Mobile subscription** | Recurring with computed quantity | **$250/month per paid subscriber** with bulk discount: every 3rd subscriber free. `paid_quantity = N − floor(N/3)`. Drives the platform's primary recurring revenue spine. |
| **Quarterly SLA fee** | Recurring quarterly | $750 / $2,500 / $7,500 (Bronze / Silver / Gold). Tiered by firm size: solo / 3-10 person / 11+ person. |
| **Monthly minimum (floor)** | Computed monthly top-up | AUD **$5,000/month per firm** ($60K/year amortized; pro-rata). Charged as `max(commit, metered + SLA + onboarding-amortization)` at end of each billing month. |
| **Founding partner discount** | Stripe coupon, year 1 only | First 10 firms to convert from trial: **50% off ALL components for 12 months** (per-claim, mobile, SLA, floor, onboarding). After 12 months reverts to standard pricing. |

### Per-claim fee tier — flat, no claim-size tiering

`$1,500 per delivery` is flat regardless of claim value ($150K minimum to $5M+). The mobile subscription scales with claimant size (more employees in claimant = more mobile subscribers), so flat per-claim doesn't create asymmetry. Larger claims drive larger mobile revenue automatically.

### Strategic context for pricing

**Financiers are the predominant generator of value over time** (Bloomberg-terminal-style two-sided market). Consultants subsidize their platform fee because financiers will pay for access to deal flow long-term. Pricing optimizes for **adoption** (land-grab while there's no competition), not for **extraction**. The federation work in Phase 3 is the rail for the future predominant revenue channel.

### Reality checks across firm shapes

**Solo consultant, 4 minimum claimants, quarterly mode**:
- Per-claim: 4 × 4 × $1,500 = $24,000
- Mobile (paid = 4 − 1 = 3): 3 × $250 × 12 = $9,000
- SLA Bronze: $3,000
- Onboarding (year 1): $5,000
- Year 1 components: $41,000 → **floor tops to $60,000**
- Year 2+: $36,000 → **floor tops to $60,000**

**Mid-tier firm, 30 claimants, quarterly mode**:
- Per-claim: 30 × 4 × $1,500 = $180,000
- Mobile (paid = 30 − 10 = 20): 20 × $250 × 12 = $60,000
- SLA Silver: $10,000
- Year 1: $5K + $250K = **$255,000**
- Founding partner Y1: **$127,500**

**Large firm, 100 claimants, quarterly mode**:
- Per-claim: 100 × 4 × $1,500 = $600,000
- Mobile (paid = 100 − 33 = 67): 67 × $250 × 12 = $201,000
- SLA Gold: $30,000
- Year 1: $5K + $831K = **$836,000**
- Founding partner Y1: **$418,000**

### Flow A: Trial → conversion → first payment → tenant activation (Phase 1)

```
[Visitor at /signup]

        ▼
[Email + firm details submitted]──▶ POST /v1/auth/signup
        │
        ▼
[Email verification (P8 T1.7)]
        │  user clicks link
        ▼
[POST /v1/auth/verify-email]
        │
        ▼
[Server creates tenant + tenant_user; trial_status='active'; ends_at=+30d]
        │
        ▼
[User logs in → trial-mode UI: "X days remaining" banner + restrictions]
        │
        ▼
[Day 23: reminder email "7 days left in your trial"]
        │
        ▼
[Day 24-30: User clicks "Upgrade to paid" CTA]
        │
        ▼
[POST /v1/billing/checkout-session]
        │
        ├─ Server checks founding_partner_slots quota
        │  If available, attach FOUNDER-XXX coupon
        │
        ▼
[Stripe Checkout opens — onboarding fee + monthly subscription pre-configured]
        │  user enters card + AU GST applies via Stripe Tax
        ▼
[Stripe processes payment]
        │
        │  ┌─── webhook: checkout.session.completed ───┐
        │  ▼                                            │
        │  [POST /v1/billing/webhook]                   │
        │  - verify signature                           │
        │  - idempotency: dedupe by event.id            │
        │  - tenant.trial_status='converted'             │
        │  - subscription created with all items        │
        │  - if FOUNDER coupon: tenant.tier=founding    │
        │                                                │
        ▼  ◀────────────────────────────────────────────┘
[Redirect /onboarding/welcome]
        │

[Tenant activation gate sees all 4 components active → grants access]
```

**Key properties:**
- Trial creates tenant rows BEFORE payment so email verification works.
- Webhook is authoritative; redirect is UX.
- Idempotency on `processed_webhook_events.stripe_event_id` PRIMARY KEY.
- Founding partner attribution happens at checkout-session creation, not at conversion.

### Flow B: Mobile subscription quantity changes (Phase 1 + 2)

```
[Firm onboards new claimant + activates mobile]
        │
        ▼
[POST /v1/claimants/:id/mobile-subscribe]
        │
        ▼
┌─── Server side ──────────────────────────────────────┐
│ - INSERT into claimant_mobile_subscription            │
│   (claimant_id, started_at=NOW)                       │
│ - UPDATE claimant.mobile_subscribed=true              │
│ - count active = COUNT(*) WHERE ended_at IS NULL      │
│ - paid_quantity = active − floor(active / 3)          │
│ - Stripe API: update subscription_item.quantity       │
│   = paid_quantity (Stripe prorates automatically)     │
└──────────────────────────────────────────────────────┘
                      │
                      ▼
[On mobile session: usage tracked separately for analytics, NOT billing]

[Firm unsubscribes claimant from mobile]
        │
        ▼
[POST /v1/claimants/:id/mobile-unsubscribe]
        │
        ▼
[Server: end claimant_mobile_subscription, recompute, sync Stripe]
```

**Key properties:**
- Bulk discount math happens on every quantity change, not at month-end.
- Stripe prorates at the day level (firm only pays for partial month if claimant added mid-period).
- `claimant.mobile_subscribed` is the canonical signal for tenant activation gate (firms with no mobile subs may still hit floor minimum).

### Flow C: Per-claim fee firing on delivery (Phase 1)

```
[Consultant transitions claim status]
        │
        ▼
[Claim.status = 'quarterly_assurance_generated' OR 'annual_claim_filed']
        │
        ▼
┌─── Server side (DB trigger or app-level handler) ────┐
│ - check claim.platform_fee_charged_at IS NULL         │
│   (idempotency on claim.id + delivery_kind)           │
│ - emit Stripe usage record:                            │
│   POST /v1/subscription_items/{id}/usage_records      │
│   { quantity: 1, timestamp: NOW, action: 'increment' }│
│ - on success: SET platform_fee_charged_at=NOW         │
│ - on failure: log, retry via pg-boss queue            │
└──────────────────────────────────────────────────────┘
                      │
                      ▼
[End of billing period]
                      │
                      ▼
[Stripe aggregates usage records → invoices firm]
```

**Key properties:**
- Per-claim fires on each delivery (quarterly + annual). A claimant with 4 deliveries/year generates 4 usage records.
- Idempotent on `(claim.id, delivery_kind)` — re-emitting Q1 doesn't double-charge.
- Firm choice (quarterly mode vs annual-only) is naturally captured by how many deliveries they create.

### Flow D: Federation token issuance + redemption (Phase 3)

```
[Consultant in firm A]
        │  goes to /admin/federation
        ▼
[Issue token UI — selects financier subject_tenant + scope + expiry]
        │
        ▼
[POST /v1/federation/tokens]
        │
        ▼
┌─── Server side ────────────────────────────────────────┐
│ - generate one-time token (256-bit random)              │
│ - INSERT delegation_token (issuing_tenant_id=A,         │
│   redeeming_subject_tenant_id=<financier>,              │
│   token_hash=SHA256(token), expires_at=+30d)           │
│ - return one-time URL: /federated/redeem?t=<token>     │
└─────────────────────┬───────────────────────────────────┘
                      ▼
[Consultant emails URL to financier (out-of-band)]
                      │
                      ▼
[Financier clicks URL]──▶ GET /federated/redeem?t=<token>
                      │
                      ▼
┌─── Server side ──────────────────────────────────────────┐
│ - lookup delegation_token by SHA256(t)                    │
│ - verify not expired, not revoked, not redeemed           │
│ - mark redeemed_at=NOW                                    │
│ - mint scoped session JWT (claim: delegation_token_id)    │
│ - set Referrer-Policy: no-referrer                        │
│ - set cookie + redirect to /federated/{tokenId}           │
└─────────────────────┬─────────────────────────────────────┘
                      ▼
[Financier lands at /federated/{tokenId}]
        │
        ▼
[Read-only mode UI loads with brand_config injected]
        │
        ▼
┌─── Each API request from financier ───────────────────┐
│ - JWT contains delegation_token_id                     │
│ - Server sets app.delegation_token_id GUC              │
│ - RLS policies check GUC + scope to allowed rows       │
│ - On every request: re-check delegation_token.revoked_at│
│   (cache-busting; ~1ms overhead, security-critical)    │
└────────────────────────────────────────────────────────┘
        │
        ▼
[Financier sees Assurance Reports + claim summaries under firm A's brand]
```

**Key properties:**
- Token at rest in DB is hashed (SHA-256). DB compromise doesn't leak active tokens.
- One-time redemption: `redeemed_at` prevents URL reuse. After redemption, token converts to a session-bound credential.
- RLS scoping uses `app.delegation_token_id` GUC, parallel to `app.current_tenant_id`.
- Brand injection is purely UI-side — server returns raw data; client reads `brand_config` from financier `subject_tenant` row.

## Error handling

Failure modes worth pre-thinking, organized by phase + cross-cutting concerns.

### 1. Phase 0 deployment risks

**`australia-southeast1` capacity** — Sydney occasionally has capacity issues for specific machine types. Detection: first `gcloud sql instances create` may fail. **Response**: fall back to `australia-southeast2` (Melbourne). Document as fallback in P9.0 runbook.

**DNS propagation lag** — Cert provisioning can take 30 min to 48 hours. Detection: cert provisioning hangs. **Response**: register DNS records 48h BEFORE Phase 0's other work, so cert provisioning is the last step.

**pgvector extension** — Must be enabled per-instance. Detection: P5/P6 tests fail. **Response**: enable via gcloud flags during instance creation; verify before running migrations.

**Migration drill** — Catch SQL surprises before production. Detection: `pnpm db:migrate` fails. **Response**: P9.0 includes a "fresh-instance migration drill" — start temp instance, run migrations, tear down.

### 2. Stripe integration risks (Phase 1)

**Webhook signature verification fails** — Wrong env var or proxy stripping headers. Detection: 400 responses on webhook hits. **Response**: log raw signature header on failure; verify Stripe dashboard URL matches.

**Webhook idempotency** — Stripe replays events. Detection: duplicate state changes. **Response**: `processed_webhook_events.stripe_event_id` PRIMARY KEY; INSERT ON CONFLICT DO NOTHING.

**AU GST registration delay** — Stripe Tax requires AU ABN; verification can take 1-3 business days. **Response**: complete in Phase 0; Phase 1 can't ship until Tax registration approved.

**Tenant activation race** — Webhook may arrive before or after redirect. **Response**: success page polls `/v1/billing/status` with 30s timeout; webhook is authoritative.

**Trial conversion edge cases** — User converts on day 31 (after expiry). **Response**: 30-day grace period after trial expiry where read-only access continues; conversion still possible. After 60 days total, archive.

### 3. Pricing edge cases (Phase 2)

**Mobile bulk discount race condition** — Two mobile-subscribe requests at same time. Detection: Stripe quantity off by 1. **Response**: serialize quantity recompute via `pg_advisory_xact_lock(hashtext('mobile_quantity_' + tenant_id))`. Same pattern as P7's `findOrCreateUser` lock.

**Floor top-up failure** — Tenant's payment method fails for top-up invoice. Detection: `floor_topup_invoice.status='failed'`. **Response**: standard dunning flow (Smart Retries + email); after final failure, suspend like other payment failures.

**Founding partner slot exhaustion** — 11th firm tries to claim. Detection: `INSERT INTO founding_partner_slots WHERE tenant_id IS NULL LIMIT 1` returns no rows. **Response**: tenant converts at standard pricing; founding partner badge not awarded.

**Per-claim fee firing for stub deliveries** — Test/staging environment generates "deliveries" that shouldn't bill. **Response**: `STRIPE_API_KEY` switches between live + test mode; usage records in test mode don't bill.

**Plan downgrade exceeds usage** — Firm on Silver SLA downgrades to Bronze mid-quarter while exceeding Bronze's volume entitlements. **Response**: validation in plan-change endpoint refuses if usage exceeds target plan; surface clear UI error.

### 4. Federation security risks (Phase 3)

**Token leak via referrer header** — Financier clicks redemption URL from external site → token in Referer header. **Response**: `/federated/redeem` sets `Referrer-Policy: no-referrer`; immediately redirects to `/federated/{tokenId}` (no token in URL). Token in URL only ~10ms before redirect.

**RLS bypass attempt** — Malicious or buggy code queries rows outside delegation scope. **Response**: P8's T1.4 RLS coverage test extends to federated session paths. Phase 3 adds tests asserting `app.delegation_token_id` set → queries return only authorized rows. Security-critical — failures are blockers, not flakes.

**`brand_config` XSS** — Consultant sets footer to `<script>alert(1)</script>`. **Response**: brand-injection layer treats `brand_config` strings as text-only. React's text rendering auto-escapes. URL fields use strict allowlist (only `https://`). No HTML allowed anywhere.

**Token revocation race** — Consultant revokes at T0; financier has open tab loaded at T0-1ms. **Response**: every authenticated endpoint reads `delegation_token.revoked_at` per request (no cache); ~1ms penalty acceptable for security.

### 5. Cross-cutting

**ISO supplier register update** — Adding GCP requires DPA review + cert evidence. **Response**: Phase 0 includes `docs/iso27001/suppliers/google-cloud.md` with cert links, DPA evidence, AU regional commitment.

**Cost runaway** — Cloud SQL HA + Cloud Run + Stripe + Resend + Sentry add up. **Response**: Phase 0 enables Budget Alerts at 50%/90%/100% of $200/month threshold; tagged per environment.

**Founding partner discount expiry** — Year 1 ends; tenant suddenly billed 2× more. **Response**: 30-day pre-expiry email warning + in-app banner; year-1 wraps up at conversion-anniversary; Stripe coupon auto-removes, full pricing applies on next billing cycle.

### What we deliberately don't test in P9

- Real financier integration (depends on actual financier conversation)
- Mobile app deep integration (separate testing in `apps/mobile/`)
- Load/scale (no production volumes warranting load testing yet)
- Multi-region failover (single-region in `australia-southeast1`)
- Long-tail Stripe webhook event types (only handle events we use)
